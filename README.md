# Car Maintenance App

Система непрерывного мониторинга техсостояния автомобиля — open-source аналог
Bluelink / FordPass, но не привязанный к конкретной марке: работает с **любой
машиной, у которой есть разъём OBD-II**.

Проект делался как дипломная работа (ВКР), поэтому акцент — на архитектуре и
современном стеке, а не на промышленной готовности. Тестовая машина — Hyundai
Creta 2020, но схема БД и API универсальны.

## Идея

К машине подключается ELM327-адаптер. Небольшой агент (`agent/obd_agent.py`)
читает с него данные по OBD-II и шлёт их на бэкенд. Дальше работают два
независимых аналитических канала:

- **Телеметрия (реактивный).** Каждый замер сравнивается с порогами по контексту
  (двигатель работает / прогрет) — и если что-то не так, создаётся `Alert` с
  уровнем серьёзности. Когда показатель возвращается в норму, алерт
  автоматически закрывается.
- **ML (предиктивный).** Модель scikit-learn (HistGradientBoosting) по 13
  признакам машины и её истории оценивает вероятность того, что скоро понадобится
  обслуживание — `risk_score`.

Каналы намеренно разделены: реальные данные с OBD не смешиваются с синтетическим
датасетом, на котором обучена модель.

Всё это видно на веб-дашборде: живые графики, активные алерты, risk_score,
история ТО. Дашборд опрашивает бэкенд раз в 5 секунд.

## Стек

**Backend:** Python 3.14, FastAPI, SQLAlchemy 2.0 (async, asyncpg), PostgreSQL,
Alembic. Аутентификация — JWT (для пользователей) и отдельные device-токены
(для агентов).

**ML:** scikit-learn, pandas, joblib. Модель обучена на Kaggle Vehicle
Maintenance Dataset.

**Frontend:** React 19 + TypeScript, Vite, Tailwind CSS v4, TanStack Query,
React Router, Recharts.

Менеджер пакетов для Python — [uv](https://docs.astral.sh/uv/).

## Структура

```
app/          ← FastAPI-приложение (models / schemas / crud / routes / services / ml)
agent/        ← obd_agent.py — агент, читает ELM327 и шлёт телеметрию
tools/         ← simulate_telemetry.py — симулятор истории для демо
migrations/   ← миграции Alembic
frontend/     ← веб-дашборд (React + Vite)
```

## Запуск

### Что нужно

- Python 3.14
- PostgreSQL
- [uv](https://docs.astral.sh/uv/) (Python-зависимости)
- Node.js 20+ (для фронтенда)

### 1. Backend

Установить зависимости:

```bash
uv sync
```

Создать файл `.env` в корне проекта (см. `.env.example`):

```
DATABASE_URL=postgresql+asyncpg://user:password@localhost:5432/car_maintenance
SECRET_KEY=любая-длинная-случайная-строка
```

Необязательные переменные (есть значения по умолчанию):

| Переменная | По умолчанию | Назначение |
|---|---|---|
| `ALGORITHM` | `HS256` | алгоритм подписи JWT |
| `ACCESS_TOKEN_EXPIRE_MINUTES` | `60` | срок жизни токена |
| `CORS_ORIGINS` | `localhost:5173` | разрешённые источники для фронта |

> `DATABASE_URL` должен содержать `+asyncpg` — приложение работает асинхронно.
> Alembic сам подменяет драйвер на синхронный (`+psycopg2`) для миграций.

Накатить миграции:

```bash
uv run alembic upgrade head
```

Запустить сервер:

```bash
uv run uvicorn app.main:app --reload
```

API поднимется на `http://127.0.0.1:8000`, Swagger — на `/docs`.

### 2. Frontend

```bash
cd frontend
npm install
npm run dev
```

Дашборд откроется на `http://127.0.0.1:5173`.

### 3. ML-модель

Готовая модель лежит в `app/ml/model.pkl` и подгружается при старте бэкенда.
Переобучить можно так (нужен датасет `vehicle_maintenance_data.csv` в корне):

```bash
uv run python -m app.ml.train
```

## Агент и симулятор

После регистрации на дашборде нужно создать машину, затем устройство — при
создании устройства модалка один раз покажет `api_token` и готовую команду
запуска агента.

**Агент** читает данные с реального ELM327 и шлёт их на бэкенд:

```bash
uv run python agent/obd_agent.py --device-id 1 --token cma_dev_...
```

Без железа под рукой — флаг `--mock` (генерирует правдоподобные данные):

```bash
uv run python agent/obd_agent.py --device-id 1 --token cma_dev_... --mock
```

**Симулятор** накатывает «месяц истории» — для демонстрации без живой машины.
Есть четыре сценария: `normal`, `dying_alternator`, `summer_overheat`,
`cold_revving` (и `all` — по сценарию на день):

```bash
uv run python tools/simulate_telemetry.py \
    --device-id 1 --token cma_dev_... \
    --scenario dying_alternator --days 30
```

## Ограничения

Это учебный проект, и часть вещей сознательно осталась за рамками:

- ELM327 не уходит в сон и сажает АКБ, поэтому агент работает **только во время
  поездок** — это ограничение бытового железа, не архитектуры.
- ML-модель обучена на синтетическом датасете, поэтому `predict_proba`
  распределён бимодально (около 0% или около 100%) — это разобрано в дипломе.
- Не реализованы: logout / refresh-токены, rate-limiting, email-верификация —
  они вынесены в направления дальнейшего развития.
